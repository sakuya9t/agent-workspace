# Client i18n conventions

The React client renders all user-facing text through i18next
(react-i18next). English is the source of truth; the daemon is not
localized (plugin display names, workspace names and server error bodies
render as-is).

## Layout

```
client/src/i18n/
  index.ts            i18next init (synchronous; no Suspense), language detection
  i18next.d.ts        typed keys: t("bad.key") fails `tsc --noEmit`
  labels.ts           enum -> label helpers (status/attention/isolation)
  time.ts             relTime()/fmtReset(), the only relative-time formatters
  locales/en.json     one file per locale, one default namespace
client/check-locales.mjs   locale-parity gate (part of lint/build)
```

## Rules

- **No literal UI strings in JSX.** Enforced by `eslint-plugin-i18next`
  (`npm run lint`, also part of `npm run build`). Escape hatch:
  `// eslint-disable-next-line i18next/no-literal-string` with a reason,
  only for genuinely untranslatable render text (e.g. commands in `<code>`).
- **Key naming:** top-level group = source file (`sessionList.*`,
  `rightPanel.*`, …); shared words live in `common.*`; enum lookups use the
  wire value verbatim (`status.running`). Semantic camelCase leaf names.
- **Call time, not module init.** Non-React code (api.ts errors, xterm
  writes, store helpers) imports the instance and calls `i18n.t()` inside
  the function that runs — never in a module-level constant, which would
  freeze one language at import time.
- **Never localize persisted data.** localStorage holds sentinels/user
  input only; display text resolves at render (see `daemonLabel()` in
  connectionStore.ts).
- **Commands, URLs and git refs stay in code**, passed to strings as
  `{{slots}}` (or `<Trans>` components when styled) so translations can't
  corrupt them.
- **Plurals** use `_one`/`_other` suffixes with `count`. A base key with
  `{{count}}` and no suffixed variants is valid and plural-ready.
- **Locale files must stay in sync.** `check-locales.mjs` (in `npm run
  lint` and the build) diffs every `locales/*.json` against `en.json`:
  missing keys, orphan keys, empty values, and `{{slots}}`/`<tags>` the en
  value doesn't have all fail. Plural variants compare as one key family,
  so a locale may carry fewer/more plural forms than English.

## Adding a locale

1. Copy `locales/en.json` → `locales/<lng>.json` and translate the values.
   `npm run lint` fails until the file is fully in sync with en.json
   (check-locales.mjs), so partial translations can't ship silently.
2. Register it in `resources` and `SUPPORTED` in `i18n/index.ts`.
3. Add a picker (e.g. in ConnectionDialog) that calls
   `i18next.changeLanguage()` and persists to `localStorage["asm.lang"]`;
   detection order is `?lang=` → `asm.lang` → default.
